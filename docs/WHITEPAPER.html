1. Introduction & Design Principles

Rez is a general-purpose, end-to-end encrypted data transport protocol built on a decentralized relay mesh with multi-hop onion routing. It provides a substrate for any application that needs to move data privately between parties — without reliance on a central authority, phone number, or identity provider.

The protocol is payload-agnostic. Any data that can be serialized into bytes — text messages, voice frames, video streams, file chunks, web pages, database replicas, IoT telemetry — can be encrypted, onion-routed, and delivered through the Rez network. The protocol layer provides identity, encryption, routing, and store-and-forward delivery. Applications built on top define what that data means.

Rez Chat, the first application built on the protocol, demonstrates the full stack: real-time encrypted messaging with contacts, groups, file transfer, and encrypted backup. But the protocol is designed to support a much broader class of applications, including private/anonymous websites, encrypted voice and video communication, distributed encrypted file backup, and any scenario where data must move between parties without trusting the infrastructure in between.

Protocol vs. Application. This white paper describes the Rez protocol — the transport, encryption, routing, and identity layers. Where application-specific features are discussed (messaging, threads, groups), they serve as concrete examples of how the protocol is used, not as the protocol's boundaries.

Core Design Principles

  • Payload-agnostic transport. The protocol encrypts and routes arbitrary byte payloads. Applications define semantics.
  • Zero trust. All data is opaque ciphertext to relay operators. Keys never leave the client device.
  • Self-sovereign identity. Accounts are Ed25519 keypairs. No phone numbers, emails, or registration servers.
  • Decentralized relay mesh. Any participant can run a relay node. The network has no single point of failure.
  • Metadata privacy. Onion routing prevents any single relay from correlating sender and recipient.
  • Forward secrecy & break-in recovery. The Double Ratchet protocol ensures that compromise of current keys cannot decrypt past or future messages.
  • Async delivery. Store-and-forward inboxes ensure data is delivered even when recipients are offline.

2.3 Layered Responsibilities

Rez enforces strict separation between the protocol and application layers. The protocol (rez-core + rez-node + rez-sdk) handles identity, encryption, routing, and delivery of opaque byte payloads. It has no concept of "messages", "threads", or "contacts" — those are application-level abstractions built on top by rez-chat or any other application.

The client SDK (rez-sdk) performs all encryption and decryption using keys that never leave the client. The relay layer (rez-node) handles routing, store-and-forward, and mesh coordination but only ever processes opaque ciphertext. The cryptographic primitives in rez-core are platform-agnostic, with concrete implementations for both Node.js (NodeCryptoProvider) and browser (BrowserCryptoProvider using WebCrypto Subtle API).

3. Identity & Key Management

3.1 Key Generation & Account Derivation

Every Rez identity is an Ed25519 signing keypair generated locally on the user's device. There is no registration server, no phone number binding, and no identity provider. The keypair is the account.

From the Ed25519 public key, Rez derives a deterministic Account ID: 

AccountID = "rez:acct:" + base32(SHA-256(publicKeyBytes))

This produces a human-readable, collision-resistant identifier like rez:acct:5k4r7xw... that uniquely identifies the account without revealing the public key directly.

For Diffie-Hellman key exchange, Rez uses X25519 keypairs. These are generated on demand for X3DH handshakes, Double Ratchet DH steps, and onion layer encryption. Every DH operation uses ephemeral keys that are discarded after use, providing forward secrecy.

3.2 Keystore Encryption

Private keys are encrypted at rest in a keystore protected by a user-chosen password. Rez supports two key derivation functions for password stretching:

KDFParametersEnvironment
PBKDF2-SHA256210,000 iterations, 32-byte outputDefault (all platforms)
scryptN=217, r=8, p=1, 32-byte outputElectron desktop (optional)

The derived 32-byte key encrypts the keystore contents with AES-256-GCM using a random 12-byte IV. The encrypted keystore envelope includes the KDF parameters, salt, and version, allowing future algorithm upgrades without breaking existing keystores.

Password buffer zeroing. After key derivation, the password buffer is immediately zeroed in memory to minimize the window of exposure.

3.3 Crypto Provider Abstraction

All cryptographic operations are mediated through an abstract RCryptoProvider interface with methods for random byte generation, SHA-256 hashing, HKDF-SHA256 key derivation, AES-256-GCM AEAD, Ed25519 signing/verification, and X25519 DH. Concrete implementations:

  • NodeCryptoProvider — Uses Node.js node:crypto module
  • BrowserCryptoProvider — Uses WebCrypto Subtle API (Chrome 113+, Firefox 128+, Safari 17.4+)

4. End-to-End Encryption

4.1 X3DH Key Agreement

Rez establishes pairwise encrypted sessions using the Extended Triple Diffie-Hellman (X3DH) protocol, adapted from the Signal Protocol specification. X3DH allows two parties to derive a shared secret even when the responder is offline, using pre-published key material.

Pre-Key Bundle

Each user publishes an X3DHPreKeyBundle containing:

  • Identity signing public key (Ed25519) — long-term identity
  • Identity-DH public key (X25519) — long-term DH key, signed by the identity key; participates in the initiator's DH1 binding so the responder's identity is cryptographically tied to the shared secret
  • Signed pre-key (X25519) — medium-term DH key, signed by the identity key
  • Signed pre-key signature — Ed25519 signature proving ownership of the signed pre-key
  • One-time pre-key (X25519, optional) — single-use key for additional forward secrecy

Initiator Flow

01 Fetch the responder's pre-key bundle
02 Verify the signed pre-key signature using the responder's identity key
03 Generate an ephemeral X25519 keypair
04 Compute four DH exchanges: DH1 = DH(initiator.identityDH, responder.signedPreKey), DH2 = DH(initiator.ephemeral, responder.identityDH), DH3 = DH(initiator.ephemeral, responder.signedPreKey), and optionally DH4 = DH(initiator.ephemeral, responder.oneTimePreKey). DH1 and DH2 bind both parties' long-term identity keys to the shared secret, preventing prekey-substitution impersonation.
05 Derive shared secret: HKDF-SHA256(DH1 || DH2 || DH3 [|| DH4], info="rez-x3dh-v2")
06 Sign the canonical handshake envelope with the initiator's Ed25519 identity key. The responder verifies this signature (and the responder's own signed-prekey and identity-DH signatures) before any state mutation, deriving the remote identity from the verified pubkey rather than trusting a self-asserted sender ID on the wire.
07 Initialize the Double Ratchet with the shared secret

Session ID

Each session is identified by SHA-256(sharedSecret), a 32-byte value that both parties can independently compute. This session ID appears in ratchet headers to allow the recipient to select the correct decryption session.

4.2 Double Ratchet

Once the X3DH shared secret is established, Rez uses the Double Ratchet algorithm for ongoing payload encryption. The protocol combines two interlocking mechanisms:

Symmetric Ratchet (KDF Chain)

Each message advances a chain key via HKDF: 

deriveMessageKey(crypto, chainKey):
  output = HKDF-SHA256(
    IKM:  chainKey,
    salt: empty,
    info: "rez-ratchet-chain-v1",
    len:  64 bytes
  )
  messageKey  = output[0..31]   // 32 bytes
  nextChainKey = output[32..63]  // 32 bytes

Each derived messageKey encrypts exactly one payload, then is discarded. The nextChainKey replaces the current chain key. Old chain keys cannot be recovered from the current state, providing forward secrecy.

Diffie-Hellman Ratchet

When a party sends a new payload, they may include a fresh X25519 public key. The recipient performs a DH exchange to derive new root key material, which reseeds the chain keys: 

deriveRootKey(crypto, rootKey, dhSecret):
  output = HKDF-SHA256(
    IKM:  dhSecret,
    salt: rootKey,
    info: "rez-ratchet-root-v1",
    len:  96 bytes
  )
  newRootKey       = output[0..31]   // 32 bytes
  sendingChainKey  = output[32..63]  // 32 bytes
  receivingChainKey = output[64..95]  // 32 bytes

This DH ratchet step provides break-in recovery (also called post-compromise security): even if an attacker obtains current key material, they cannot decrypt payloads sent after the next DH ratchet step.

Message Key → AEAD

Each message key is expanded into an AES-256-GCM key and nonce: 

deriveAeadKeyNonceV1(crypto, messageKey, sid, pn, n, dh):
  HKDF-SHA256(
    IKM:  messageKey,
    salt: empty,
    info: "rez-aead-v1",
    len:  44 bytes
  )
  aeadKey = output[0..31]   // 32 bytes
  nonce   = output[32..43]  // 12 bytes

4.3 Encrypted Envelope Wire Format

Each encrypted payload is serialized as an EncryptedEnvelopeV1: 

{
  "v": 1,
  "suite": "HKDF-SHA256/AES-256-GCM",
  "header": {
    "v": 1,
    "sid": "<32-byte session ID, base64>",
    "dh":  "<X25519 public key bytes, base64>",  // null if no DH ratchet
    "dhAlg": "X25519",
    "dhFmt": "raw",
    "pn": 5,     // previous chain length
    "n":  0      // message index in current chain
  },
  "nonce": "<12-byte IV, base64>",
  "ct":    "<ciphertext + 16-byte GCM tag, base64>"
}

The header is sent in plaintext (it contains no secret material) and allows the recipient to advance their ratchet state to the correct position before attempting decryption. Skipped keys are cached for out-of-order delivery.

5. Onion Routing

While E2EE protects content, it does not hide who is communicating with whom. Rez addresses this with multi-hop onion routing, inspired by Tor but purpose-built for asynchronous data transport rather than real-time streaming.

5.1 Path Construction

The sender's node selects 1–3 relay hops from the known relay pool using the GatewayRelaySelector. Relays are chosen uniformly at random from the set of eligible relays (those with valid, non-expired onion keys and reachable TCP endpoints). The entry relay knows the sender but not the recipient. The delivery relay knows the recipient's inbox but not the sender. Any intermediate relays know neither.

Figure 2 — Three-hop onion routing path

5.2 Packet Encoding (Backward Layering)

Onion packets are constructed from the innermost layer outward. Starting with the plaintext payload, each hop's layer is encrypted in reverse order:

01 Start with blob = innerBytes (the E2EE-encrypted message)
02 For each hop i from last to first:
  • Generate ephemeral X25519 keypair
  • Compute shared secret: DH(ephemeral.priv, relay[i].onionPubKey)
  • Derive AEAD key + nonce: HKDF-SHA256(sharedSecret, info="rez-onion-v2" || hopIndex)
  • Build layer plaintext with routing directives (next, ttl, deliver at final hop)
  • Encrypt with AES-256-GCM, using canonical JSON of {v, hopIndex, ttl, onionKeyId} as AAD
  • Set blob = cipherJSON(ephPub, ct, onionKeyId)
03 Select the smallest size class that fits the blob (4KB, 8KB, 16KB, 32KB, 64KB, 128KB, or 256KB)
04 Pad blob with zeros to exactly the size class boundary
05 Wrap in OnionPacketV2 { v: 2, sizeClass, payload }

Layer Plaintext Structure

{
  "v": 2,
  "ttl": 3,                              // decremented at each hop
  "next": { "relayKeyId": "relay_xyz" },  // next hop's identity
  "flags": { "dropOnFail": true },        // drop silently on routing failure
  "inner": "<base64-encoded encrypted inner blob>",
  // Final hop only:
  "deliver": { "inboxId": "inbox:hosted:abc123" },
  "receipt": { "inboxId": "inbox:hosted:sender_inbox" },
  "returnPath": { ... }                   // SURB for delivery receipts
}

5.3 Relay-Side Peeling

When a relay receives an onion packet, the OnionPeelCodecV2 processes it:

  1. Hash the payload to compute packetIdHex = SHA-256(payload)
  2. Parse the outer JSON layer to extract the cipher object (ephPub, ct, onionKeyId)
  3. Replay check: verify packetIdHex:hopIndex:onionKeyId has not been seen before (FIFO cache, max 50,000 entries)
  4. Look up the private key for onionKeyId in the relay's keyring (validates timing windows and key status)
  5. Compute shared secret from relay's private key + ephemeral public key
  6. Derive AEAD key/nonce and decrypt with AES-256-GCM
  7. Validate TTL > 0 (drop if expired); decrement TTL by 1
  8. If deliver directive present: deposit to inbox. Otherwise: forward inner blob to next hop.

5.4 Traffic Analysis Resistance

All onion packets are padded to one of seven fixed size classes:

Size ClassBytesTypical Use
4 KB4,096Small payloads, control frames
8 KB8,192Text payloads, small metadata
16 KB16,384Larger payloads, invite bindings
32 KB32,768Multi-part payloads, media frames
64 KB65,536File transfer chunks
128 KB131,072Large file chunks
256 KB262,144Maximum single packet (max 1 MB total)

An observer sees only fixed-size packets traversing the network, making it impossible to correlate packets by size as they pass between relays.

5.5 Onion Key Management

Each relay maintains a keyring of X25519 onion keys with defined validity windows (notBefore, notAfter) and lifecycle statuses:

  • active — preferred for new packets
  • draining — still valid for decryption, not used for new encryption
  • revoked — permanently disabled

Key rotation is published via updated RelayDescriptorV1 records. The overlapping validity windows ensure packets encrypted with a recently-rotated key can still be decrypted during the transition.

6.2 Peer Authentication

When two relays establish a TCP connection, they perform mutual challenge-response authentication:

  1. peer.hello — connecting relay sends its node key ID and public key
  2. peer.challenge — receiving relay issues a 32-byte random nonce with 60-second TTL
  3. peer.identify — connecting relay signs the challenge payload (nonce + both key IDs) with Ed25519
  4. peer.accept — receiving relay verifies signature, assigns trust level
  5. peer.bind — both sides exchange signed relay descriptors

Single-use challenges. Each nonce is consumed on first use and tracked in a consumed-challenge set with TTL-based expiry (60 seconds), preventing replay attacks.

6.3 Mesh Discovery & Gossip

The MeshCoordinator manages relay discovery and mesh health:

  • Seeded gossip: bootstrap from configured seed directory endpoints
  • Descriptor exchange: peers share descriptors on connection
  • Periodic refresh: every 30 seconds, evict expired descriptors and discover new peers
  • Peer limits: minimum 3, maximum 32 concurrent peer connections
  • Health tracking: each peer rated as healthy, degraded, or stale

6.4 TCP Framing

Inter-relay communication uses length-prefixed TCP frames: 

[4 bytes big-endian length][payload (length bytes)]
Maximum frame size: 32 KB + 4 bytes header

Rate limiting: 200 frames/second per relay socket. Excess frames are silently dropped. Per-connection flood gating: 100 frames/second with 200 burst capacity.

7. Inbox Discovery & Data Routing

7.1 Hosted Inbox Registry

Each Rez node hosts inboxes for the accounts it serves. The HostedInboxRegistry derives a stable, deterministic inbox ID for each account: 

localInboxId = "inbox:hosted:" + sanitize(accountId)

Inbox registrations include a signed delegation proof — the account's identity key signs a statement authorizing the node to host its inbox: 

{
  "kind": "hosted-inbox-delegation",
  "accountId": "rez:acct:...",
  "inboxId": "inbox:hosted:...",
  "nodeKeyId": "nk_1",
  "nodePublicKeyB64": "...",
  "issuedAtMs": 1709251200000,
  "expiresAtMs": 1711929600000
}

7.2 Inbox Routing Table

The InboxRouter maintains a distributed routing table mapping inbox IDs to delivery targets. Each route entry specifies:

  • Direct: inbox owner connected to this relay via socket — deliver locally
  • Remote: inbox reachable via next-hop relay — forward via ID-based routing

7.3 Routing Control Protocol

Control MessageDirectionPurpose
inbox.registerNode → RelayRegister locally-hosted inbox IDs with delegation proofs
inbox.routeRelay ↔ RelayAnnounce reachable inbox routes with hop count
inbox.withdrawAnyWithdraw routes (on disconnect or explicit removal)
inbox.queryRelay → PeerAsk peers if they know a route to a specific inbox
inbox.query.replyPeer → RelayRespond with route information
inbox.depositAnyDeliver encrypted payload to inbox

7.4 Deposit Flow

When the final relay in an onion path peels its layer and finds a deliver directive, it processes the deposit:

  1. Look up inboxId in the routing table
  2. If direct + socket connected: send inbox.deposit control message to the connected node
  3. If direct + no socket: persist to the local inbox store (store-and-forward)
  4. If remote: forward to the next-hop relay via the peer connection

Recipients retrieve stored payloads via the mailbox.list, mailbox.fetch, and mailbox.ack protocol commands when they reconnect.

Discovery vs. Delivery Paths

A critical distinction: the discovery path (how the route was learned) is different from the delivery path (the onion route chosen for the message). Route discovery propagates through the mesh's control plane, while delivery uses a freshly-selected onion path through randomly chosen relays. This prevents an adversary who compromises a discovery hop from predicting the delivery route.

9. Client Architecture

9.1 Session Authentication

Clients authenticate to their home node via a three-step handshake over WebSocket:

  1. session.hello — client sends account ID, identity public key, device ID, client version
  2. session.challenge — node responds with random 32-byte nonce (60s TTL), node identity, X3DH subkey
  3. session.authenticate — client proves two things:
    • Session auth signature: signs the challenge nonce together with the node's identity (its nodePublicKeyB64 and relayKeyId) using the account's Ed25519 identity key. Including the node identity in the signed payload prevents a cross-node MITM from replaying a challenge from node A to node B and tricking the client into authenticating to the attacker's node.
    • Inbox delegation: signed statement authorizing the node to host the account's inbox
    Peer-link establishment (the X3DH handshake described in §4.1) happens separately and is bound to the initiator's long-term identity key through DH1, independent of the session-auth layer.
  4. session.ready — node confirms authentication, returns session ID and local inbox ID

9.2 Uplink Pool & Failover

The UplinkPool manages WebSocket connections to one or more Rez nodes, providing automatic failover and reconnection with exponential backoff. The pool abstracts multi-node connectivity, allowing clients to maintain service even when individual nodes are unreachable.

9.3 Wire Protocol

Client-node communication uses JSON frames over WebSocket. The frame format is application-agnostic — the body carries arbitrary payload: 

{
  "id": "message.send:1709251200000:1",  // type:timestamp:sequence
  "t": "message.send",                    // message type
  "v": 1,                                 // protocol version
  "body": { ... }                          // type-specific payload
}

10.2 Integrity Verification

On reassembly, the recipient verifies:

  1. Each chunk's SHA-256 hash matches the corresponding entry in chunkHashesHex
  2. All chunks are present (no missing indices)
  3. The reassembled file's SHA-256 matches fileHashHex

Any hash mismatch causes the transfer to be rejected, preventing tampering or corruption.

11. Encrypted Storage at Rest

All persistent data on a Rez node is encrypted at rest using AES-256-GCM. The encryption key is derived from the node's Ed25519 private key, ensuring that only the key holder can read stored data.

11.1 Key Derivation

storageKey = HKDF-SHA256(
  IKM:  nodePrivateKeyBytes,
  salt: "rez:storage:encryption:v1",
  info: "rez:kv:aes256gcm",
  len:  32 bytes
)

11.2 Storage Architecture

Three decorator stores provide transparent encryption over the base storage layer:

DecoratorBase StoreData Type
EncryptedObjectStoreFsObjectStorePublished objects, blobs
EncryptedMailboxStoreFsMailboxStoreInbox deposits, mailbox events
EncryptedKeyValueStoreFsKeyValueStoreContacts, threads, groups, invites

11.3 Progressive Migration

When encryption is enabled on an existing node, Rez performs progressive migration: legacy plaintext data is read normally and re-encrypted on the next write. This avoids a costly bulk migration and allows encryption to be enabled non-disruptively.

Bootstrap exception. Only config.json and the bootstrap identity read are stored unencrypted — these are required to derive the encryption key itself.

The backup system stores client-encrypted artifacts on the node:

  • Client encrypts backup data before upload — node stores opaque ciphertext only
  • Metadata minimization: only accountHash, type, seq, createdAtMs, expiresAtMs, and sizeClass are visible
  • Default retention: 90 days
  • Restore flow: fetch checkpoint list → download latest checkpoint → fetch deltas → decrypt locally

13. Applications on Rez

The Rez protocol is a transport substrate, not a messaging application. It provides encrypted, onion-routed, store-and-forward delivery of arbitrary byte payloads between cryptographic identities. Applications built on Rez inherit all of the protocol's privacy, security, and decentralization properties without reimplementing them.

13.1 Rez Chat — Encrypted Messaging (Shipped)

The first application built on the protocol. Rez Chat provides real-time encrypted messaging with contacts, groups, threads, channels, file transfer, link previews, and encrypted backup. It demonstrates the full protocol stack:

  • Peer linking via cryptographic invite codes carrying X3DH pre-key bundles
  • Threads with deterministic IDs derived from participant identities
  • Group messaging with role-based membership (admin/member) authorized by signed member.join operations in GroupOpPayloadV1, with per-member pairwise E2EE
  • Channels — named persistent sub-conversations within groups, with per-channel read cursors and catchup
  • File transfer via chunked, hash-verified payloads routed as standard onion packets
  • Link previews generated client-side; the relay never sees the URL
  • Inbox catchup — cursor-based replay of missed messages when a client reconnects after an offline period; reads buffered deposits the relay held for the user
  • Encrypted backup where the node stores only opaque ciphertext

13.2 Private Websites & Hidden Services (Planned)

The onion routing layer can serve content anonymously. A Rez node hosting a website would publish its inbox ID as a hidden service address. Visitors send HTTP-like request payloads through the onion network; the server responds through a SURB (single-use reply block) return path. Neither the visitor nor any relay learns the server's network location.

Unlike Tor hidden services, Rez hidden services operate over the same relay mesh used for messaging, requiring no separate circuit infrastructure. The protocol's store-and-forward inboxes naturally handle request queuing, and the fixed-size packet classes prevent traffic analysis of page sizes.

13.3 Encrypted Voice & Video (Planned)

Real-time media can be transported over Rez by treating audio/video frames as a stream of encrypted payloads. The protocol's onion routing adds latency, making it best suited for privacy-prioritized communication rather than low-latency conferencing. Key properties:

  • Each voice/video frame encrypted with the existing Double Ratchet session
  • Frames routed as onion packets through the relay mesh
  • Fixed-size packet classes prevent frame-size fingerprinting
  • No centralized TURN/STUN servers required — relays handle NAT traversal implicitly

13.4 Distributed Encrypted Backup (Planned)

The protocol's encrypted storage and relay mesh enable a decentralized backup system. Users can distribute encrypted backup shards across multiple Rez nodes, where each node stores only opaque ciphertext. Key characteristics:

  • Backup data encrypted client-side before distribution — no node can read contents
  • Shards distributed across geographically diverse relay nodes for redundancy
  • Retrieval via onion routing preserves privacy of which backup belongs to whom
  • The protocol's existing integrity verification (SHA-256 per chunk) ensures shard correctness
  • No centralized storage provider — any Rez node operator can contribute storage

13.5 Building on the Protocol

Any application can build on Rez by using the rez-sdk to:

  1. Generate an identity — Ed25519 keypair, no registration required
  2. Establish encrypted sessions — X3DH + Double Ratchet with any peer
  3. Send arbitrary payloads — encrypt, onion-wrap, and deposit to any inbox
  4. Receive payloads — retrieve from inbox, decrypt, process application-specific data

The protocol imposes no schema on payload contents. An application can define its own frame types, serialization formats, and interaction patterns while inheriting E2EE, onion routing, metadata privacy, and store-and-forward delivery from the transport layer.

Key Differentiators

  • vs. Signal: Signal is a messaging application with a fixed feature set and centralized infrastructure. Rez is a protocol — it provides the same Double Ratchet E2EE but adds onion routing for metadata privacy and a decentralized relay mesh. Signal's server sees who talks to whom and when. Rez's relays see only opaque onion packets. Crucially, Rez can transport any payload type, not just chat messages.
  • vs. Matrix: Matrix is a federated communication protocol, but it is tightly coupled to its room/event data model. Homeservers store message history in plaintext (or optionally encrypted) and see full metadata. Rez nodes store only ciphertext, use onion routing for metadata privacy, and provide per-message forward secrecy via Double Ratchet (vs. Matrix's session-level Megolm). Rez's transport layer is data-model agnostic.
  • vs. Briar: Briar provides strong privacy via Tor but is tightly coupled to its messaging/forum model and requires both parties to be online simultaneously (without the mailbox plugin). Rez provides comparable metadata protection with the convenience of store-and-forward delivery and extensibility to non-messaging applications like hidden services and encrypted backup.
